# Tighten

收紧技术文档，让每个字都有用。技术文档的读者（人或 AI 代理）是带着明确目的快速扫读，废话降低信噪比、抬高定位成本。

## Core Stance

这是 smell 目录，不是从头跑到尾的清单。用它识别废话类型，再逐处做判断。

- **过度删减 = 失败**，等于不删。一句已经自然、清晰、稳定的话，留着。多数精简是做减法（删重复、删结论式重述、删画蛇添足），不是逐句替换。
- **六类废话是例子，不是查找替换。** 命中但语境里读着自然的，保留。匹配 smell，不是匹配字符串。
- **作者的声音优先。** 保留作者现有的用词、节奏、立场。规则是默认值，不是法律。
- 改完不重读全文验证——Edit 已保证 `old_string` 精确匹配。

## Pre-flight

1. **通读全文**（cat 完整文件，不要只看片段）。碎片化阅读会漏掉重复。
2. **多文档先定范围**：项目若有 README + AGENTS.md + CLOUDFLARE.md，先确认每份各管什么。面向用户的文档（README）不该塞运维细节，留指针即可。

## Hard Rules

- **命令、代码、踩坑表、故障排查、正反例**：永远保留，这是文档的核心价值。
- **不重组结构、不改内容范围**。缺关键内容是「补充」不是「精简」，告知用户即可，别顺手加。
- **语言层只作用于自然语言正文**，绕过代码字面量、JSON 键名、URL、API 路径、数据库字段名、表头里镜像接口定义的标签。
- **不解释显而易见的事**，但要解释复杂的事。规则不明显时，原因帮助执行（如「PowerShell 脚本不能写中文——编码会损坏解析」）。废话是解释显而易见，不是解释复杂。

## 六类废话（结构层，逐类排查）

对每个句子问一句：**「删掉它，读者会少得到任何信息吗？」**

### 1. 自我介绍式

解释文档「是给谁看的」「读完能怎样」，而不是直接给内容。

❌ 本文件为 AI 编码代理提供项目上下文。阅读本文件后即可上手，无需翻阅历史对话。
✅（删掉。文档内容自己会说话。）

### 2. 画蛇添足

标题/结构已经表达的事，再用一句话强调。

❌ ## 安全约束（严格遵守）
   ⚠️ 任何会话必须遵守以下红线：
✅ ## 安全约束（严格遵守）
   （直接进入列表，标题已经说明了一切）

### 3. 元废话

描述「本文档的写法/原则」，而不是文档内容本身。

❌ 本文档写的是操作方法，不写「已登录」等瞬态状态——换台电脑就不同了。
✅（删掉。直接用无状态的写法即可，不用宣告你的写法。）

### 4. 重复

同一信息在文档内多处出现。

❌ 表格里写 `固定访问地址 | xxx（永久不变，重启电脑地址也不变）`
   正文里又写 `经命名隧道，重启电脑地址不变`
✅ 只保留一处，删掉次要位置的重复。

### 5. 重复罗列

指针链接后面，把目标文档的章节目录又抄了一遍。

❌ 详见 CLOUDFLARE.md（认证查询、登录、隧道命令、从零配置、故障排查）
✅ 详见 CLOUDFLARE.md
   （目标文档自己有目录，不用替它抄。）

### 6. 过度解释

把显而易见的事说破。

❌ ⚠️ 三级边界，操作前必读。违反会导致数据丢失或收费/安全风险。
✅ ⚠️ 三级边界，操作前必读。
   （「会导致风险」是「操作前必读」的显然推论。）

## 语言层精炼（顺手做）

删完废话后，对可见正文做这几件事。**下列规则主要面向中文文档**；英文文档只做结构层精简，不强行套中文留白/引号/易错词规则。

### 排版

- **中英文/数字之间留白**：可见正文里，中文与可读的半角英文、缩写、独立数字、版本号之间加空格。
  - `获取批量 ID` / `HTTP 请求` / `版本 2.0` / `与 PM 协作` ✅
  - `user_id` / `/api/example` / `statusCode` / `Content-Type`（在代码里）→ **保持原样，不加空格**
  - 边界不清时优先保持原样，人工判断，不机械替换。
- **引号统一为直角引号** `「」`，不用中文双引号 `“”`。
- 避免感叹号、连续省略号、口号式断句。

### 称呼（作用于被精简的文档正文）

- 不用 `你` / `您` / `同学`。用 `开发者` / `实施人员` 等具体角色名，或用无主句。
- 避免问候式开场（`Hello` / `Hi`）和宣传腔（`领先` / `强大` / `重磅` / `颠覆` / `震惊`）。

### 易错表达（常见高频）

优先写成具体、可验证、不歧义的表达：

- `登陆系统` → `登录系统`
- `截止 4 月 12 日` → `截至 2026 年 4 月 12 日`（用 `截至`，给完整日期）
- `阀值` → `阈值`；`配制` → `配置`；`布署` → `部署`；`起用` → `启用`
- `完成授权`（指身份校验时）→ `完成认证`（认证 = 确认身份，授权 = 授予权限）
- `尽快处理` → `30 分钟内处理`（模糊时限改明确）
- `模型出现幻听` → `模型出现幻觉`

### 数量表达（逻辑易错）

- `缩小了 3 倍` → `缩小到原来的 1/3` / `减少了 2/3`
- `转化率提升了 5%`（20% → 25%）→ `提升了 5 个百分点` / `相对提升 25%`
- `预计大约在 3 点左右` → `预计 3 点`（去掉叠床架屋的程度副词）

## 怎么做

1. **逐段标记**废话位置和类别（用上面六类对照）。
2. **批量改**：用 Edit 逐处删除或精简。
3. **扫描确认**：用一次性正则/关键词扫描，确认常见废话模式（「无需翻阅」「违反会导致」「任何会话必须遵守」等）已清除。grep 在某些 shell（如 Git Bash）有兼容问题，用 Python 正则更可靠。

## Gotchas

| 做过什么 | 对应规则 |
|---|---|
| 把六类废话当查找替换清单，机械命中就删 | 是 smell 目录不是清单；语境里自然的保留 |
| 顺手重组标题顺序或合并章节 | 不重组结构，除非用户明确要求 |
| 删了「为什么」连同规则一起删掉 | 复杂的事要解释，只删显而易见的事 |
| 对代码里的 `user_id` / `/api/...` 加中英文空格 | 语言层只碰自然语言正文，绕过代码 |
| 删完附了一份变更清单 | 输出是改后的文本，不附说明 |

## Output

返回改后的文本。除非用户明确要「改了哪些」「review 注释」，否则不附变更清单、不解释改动、不加结尾套话。

## 维护规则

新教训折进现有原则，不许单调地加废话类型。把具体案例收回到六类之一，比新增第七类更对。
